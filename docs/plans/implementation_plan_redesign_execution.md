# /docs/design/implementation_plan_redesign_execution.md

## 목적

이 문서는 redesign 방향을 실제 구현으로 옮기기 위한 실행 계획서다.
제품 기준은 `../design/game_design.md`이고, 이 문서는 현재 레포의 구형 액션 구조를 어떤 순서로 걷어내고 새 씬/피처 흐름으로 갈아탈지 정의한다.

---

## 결론: 권장 마이그레이션 방식

권장 방식은 **기존 `JG_GameScene`를 직접 수술하지 않고, 새 전투 씬을 병렬로 만든 뒤 컷오버하는 방식**이다.

이유:

- 현재 `JG_GameScene`는 `GameSceneRoot` 아래에서 Player/Skill/Wave/Combat/Status가 구형 액션 루프 기준으로 강하게 결합돼 있다.
- `Player`는 이동/점프/다운/구조/마나까지 품고 있고, `Skill`은 시작 스킬 선택/덱 순환/스킬바를 전제로 한다.
- 이 구조 위에서 부분 치환을 하면 구형 개념과 신규 개념이 한 씬에 공존하게 되어 디버깅 비용이 급격히 커진다.

따라서 구현 전략은 아래와 같다.

1. `JG_LobbyScene`는 유지하고, 편성 진입 흐름만 추가한다.
2. 새 전투 씬을 별도로 만든다. 권장 이름은 `JG_BattleScene_Redesign`.
3. 새 씬에서 `Player/Skill/Wave/Combat` 책임을 새 방향으로 재구성한다.
4. 로비의 시작 경로를 새 씬으로 바꾼다.
5. 컷오버가 끝난 뒤 기존 `JG_GameScene`와 구형 기능을 정리한다.

---

## 씬 전략

### 유지

- `Assets/Scenes/JG_LobbyScene.unity`
  - Photon 로비/룸 생성/입장 구조는 재사용한다.
  - 다만 게임 시작 직전 흐름은 `즉시 전투 시작`이 아니라 `편성 확인 -> 전투 시작`으로 바꾼다.

### 신규 생성

- `Assets/Scenes/JG_BattleScene_Redesign.unity`
  - 공용 코어 방어
  - 반개방 맵
  - 유닛 슬롯 기반 소환
  - 에너지 기반 전투
  - 앵커 반경 기반 자동 교전

### 최종 정리

- 새 씬이 메인 전투 씬으로 검증되면:
  - 로비에서 이 씬으로 이동하도록 고정
  - 기존 `JG_GameScene`는 아카이브 또는 삭제 후보로 전환

직접 덮어쓰는 것보다 병렬 씬을 권장하는 이유는 **실패 시 되돌리기와 비교 검증이 훨씬 쉽기 때문**이다.

---

## 구현 원칙

1. 새 시스템은 구형 시스템의 옵션 분기나 임시 플래그로 만들지 않는다.
2. 새 씬에서는 `mana`, `start skill selection`, `deck rotation`, `friendly fire`, `down/rescue`를 전제하지 않는다.
3. 구형 피처 폴더를 재사용하더라도, 책임은 새 SSOT 기준으로 재정의한다.
4. 씬 계약은 실제 씬이 생기는 즉시 해당 feature README와 함께 갱신한다.
5. 컷오버 전까지는 "구형 씬은 구형 규칙", "신규 씬은 신규 규칙"으로 분리한다.

## 대상별 재정의

### Lobby

기존 책임 중 유지:

- 룸 생성/입장/퇴장
- 팀 구성
- Ready
- 시작 트리거

새로 추가할 책임:

- 편성 진입
- 각 플레이어의 편성 완료 상태 표시
- 전투 시작 시 새 씬 로드

제거 또는 축소할 것:

- "시작하면 곧바로 시작 스킬 선택으로 간다"는 전제

### Player

현재 `Player`는 아바타, 이동, 점프, HP, 다운/구조, 마나를 모두 가진다.
redesign에서는 아래처럼 줄인다.

- 플레이어 소유자 ID
- 에너지 상태
- 편성 정보
- 전투 중 소환 권한

즉 `월드 캐릭터`보다는 `플레이어 전투 세션 상태`에 가까워진다.

### Skill

현재 `Skill`은 덱 순환과 직접 시전을 중심으로 설계돼 있다.
redesign에서는 두 갈래로 분해한다.

- 유닛/모듈 데이터 카탈로그
- 전투 중 유닛 슬롯/소환 요청

즉 현재 `Skill` 피처는 장기적으로 이름을 유지하더라도, 실제 의미는 `Ability/UnitLoadout` 쪽으로 재편될 가능성이 크다.

### Combat

현재 `Combat`은 투사체/존/FF 기반 데미지 파이프라인에 가깝다.
redesign에서는 아래 중심으로 바뀐다.

- 유닛 배치 가능 판정
- 소환 처리
- 유닛 전투 판정
- 코어 피격/파괴 판정

### Wave

현재 `Wave`는 카운트다운 + 웨이브 클리어 + 보상 선택 루프다.
redesign에서는 아래로 바뀐다.

- 연속 압박 관리
- 적 조합/진입 관리
- 공용 코어 방어 상태
- 전투 종료 판정

`웨이브 보상 선택`은 제거 대상이다.

## 단계별 실행 계획

### Phase 0. 컷오버 기준 고정

목표:

- 구형 시스템을 더 확장하지 않도록 작업 기준을 먼저 고정한다.

작업:

- `../design/game_design.md` 기준 용어를 팀 공용 기준으로 확정
- `implementation_plan_redesign_migration.md`와 이 문서를 기준으로 작업 순서 고정
- 새 작업에서 `마나`, `덱 순환`, `다운/구조`를 핵심으로 삼지 않도록 합의

완료 기준:

- 문서 기준에서 새 루프가 SSOT로 인식될 것
- 신규 작업 요청이 오면 어느 씬/피처를 기준으로 작업할지 혼선이 없을 것

### Phase 1. 로비에서 편성 단계 분리

목표:

- 기존 로비에서 바로 전투 씬으로 넘어가던 흐름을 끊고, `편성`을 독립 단계로 만든다.

권장 방식:

- `JG_LobbyScene` 안에 `Garage/Loadout` UI 구역을 추가하거나
- 로비 내 패널 기반으로 편성 플로우를 먼저 구현한다

이 단계에서 필요한 것:

- 플레이어별 3~5기 편성 데이터 구조
- 편성 완료 여부
- 룸 내 다른 플레이어의 편성 준비 상태 표시
- Master 기준 시작 가능 조건 재정의

보류:

- 실제 전투 씬 구현
- 유닛 AI 세부 구현

완료 기준:

- 로비에서 "방 입장 -> Ready -> 편성 확인 -> 시작"이 자연스럽게 이어질 것
- 기존 "시작 스킬 선택" UI로 넘어가는 경로가 끊길 것

### Phase 2. 새 전투 씬 뼈대 생성

목표:

- 구형 `JG_GameScene`와 분리된 신규 전투 씬의 최소 골격을 만든다.

씬 구성 권장:

- `BattleSceneRoot`
- `WorldRoot`
- `UIRoot`
- `ServiceRoot`

월드 최소 요소:

- Core 오브젝트
- 반개방 지형
- 소환 가능 구역
- 적 진입 경로

UI 최소 요소:

- 에너지 HUD
- 유닛 슬롯 HUD
- 코어 HP HUD
- 결과 화면

서비스 최소 요소:

- 전투 세션 bootstrap
- 플레이어 소유 편성 로더
- 적 압박 스포너
- 네트워크 동기화 루트

완료 기준:

- 로비에서 새 씬을 열 수 있다
- 아직 전투가 미완이어도, 플레이어/코어/UI가 기본 골격으로 올라온다

### Phase 3. 플레이어 상태를 소환자 모델로 재구성

목표:

- 직접 조작 캐릭터 모델에서 벗어나, 플레이어를 전투 중 소환자 상태로 바꾼다.

작업:

- 이동/점프 입력 제거 또는 비활성화
- 플레이어당 에너지 상태 모델 추가
- 플레이어당 편성 슬롯/로스터 보유
- 전투 중 소환 요청 권한 모델 추가

핵심 결정:

- 플레이어를 월드에 보이는 아바타로 남길지 여부

권장:

- 첫 버전은 **월드 아바타를 제거하거나 최소화**한다
- 필요하다면 "지휘 마커" 수준의 약한 표현만 둔다

완료 기준:

- 플레이어 기본 입력이 이동이 아니라 슬롯 선택/배치가 된다
- 전투 세션 상태가 `hp/mana/downed`보다 `energy/loadout/summon authority` 중심으로 읽힌다

### Phase 4. Skill 피처를 Unit/Garage 피처로 치환

목표:

- 스킬 카탈로그와 덱 순환을 유닛/모듈/편성 구조로 바꾼다.

권장 작업 순서:

1. 데이터 레벨에서 `SkillData`에 해당하던 역할을 `UnitFrame + Module` 조합으로 재정의
2. 시작 스킬 선택 UI 제거
3. 스킬바 2슬롯을 유닛 슬롯 UI로 대체
4. 덱/드로우/보상 선택 로직 제거

중요:

- 이 단계에서는 기존 `Skill` 폴더를 바로 이름 변경하지 않아도 된다
- 대신 README와 클래스 책임부터 새 의미로 바꾼다

완료 기준:

- 전투 진입 시 플레이어는 유닛 3~5기 편성을 가진다
- 전투 중 선택 대상은 스킬이 아니라 유닛 슬롯이다

### Phase 5. 소환과 앵커 반경 전투 구현

목표:

- redesign의 핵심 전투 규칙을 처음으로 실제 플레이 가능한 수준으로 만든다.

작업:

- 배치 가능 영역 판정
- 유닛 소환 처리
- 유닛별 앵커 위치 설정
- 앵커 반경 내 자동 교전
- 코어를 향한 적 압박

첫 버전에서 하지 않을 것:

- 전투 중 재배치
- 웨이포인트
- 복잡한 역할 AI
- 플레이어 직접 스킬 시전

완료 기준:

- 플레이어가 유닛을 언제 어디에 낼지 고민하게 된다
- 유닛 역할 차이가 앵커 반경/사거리/유지력으로 읽힌다

### Phase 6. Wave를 연속 압박 구조로 교체

목표:

- 기존 웨이브 클리어 후 보상 선택 루프를 제거하고, 공용 코어 방어 중심 구조로 교체한다.

작업:

- 보상 선택 UI 제거
- 시작 스킬 선택 후 콜백 초기화 제거
- 연속 압박 또는 단계형 압박 로직으로 전환
- 적 역할 조합 중심의 스폰 구조 설계

권장 최소 적 역할:

- 러셔
- 탱커
- 후열 위협형
- 군집형

완료 기준:

- 전투 흐름이 `전투 -> 보상 선택 -> 다음 웨이브`가 아니라
  `압박 대응 지속 -> 성공/실패 종료`로 읽힌다

### Phase 7. 로비 -> 새 전투 씬 컷오버

목표:

- 실제 플레이 경로를 구형 씬에서 신규 씬으로 바꾼다.

작업:

- `Lobby`의 게임 시작 경로를 `JG_BattleScene_Redesign`로 변경
- QA/플레이테스트 기본 경로 변경
- 개발용 스모크 테스트 경로 업데이트

컷오버 조건:

- 편성 UI가 최소 동작
- 전투 씬이 최소 루프를 수행
- 코어 방어 승패가 동작
- 네트워크 기준 2인 이상 동기화가 깨지지 않음

### Phase 8. 구형 시스템 정리

정리 대상:

- 시작 스킬 선택 UI
- 덱 순환
- 마나 HUD
- 프렌들리 파이어 피드백
- 다운/구조/bleedout
- 웨이브 보상 선택

정리 원칙:

- 컷오버 전에는 삭제하지 않는다
- 컷오버 후에는 SSOT와 충돌하는 구형 규칙부터 제거한다

## 씬 컷오버 체크리스트

새 전투 씬을 기본 경로로 바꾸기 전에 아래를 만족해야 한다.

- 로비에서 2~4인 입장 후 편성 완료 상태가 보인다
- 시작 시 새 전투 씬으로 이동한다
- 각 플레이어가 유닛 슬롯과 에너지를 가진다
- 공용 코어가 존재하고 패배 조건이 동작한다
- 적이 연속 압박을 만든다
- 유닛 소환 위치와 타이밍이 실제 판단이 된다
- 결과 화면 후 로비 또는 재시작 흐름이 정리된다

## 추천 구현 순서 요약

가장 안전한 순서는 아래다.

1. 로비에 편성 단계 추가
2. 새 전투 씬 생성
3. 플레이어 상태를 소환자 모델로 전환
4. 스킬 시스템을 유닛/모듈/슬롯 시스템으로 전환
5. 소환/앵커 전투 구현
6. 웨이브 보상 루프 제거
7. 로비 시작 경로 컷오버
8. 구형 씬과 시스템 정리

## 첫 스프린트 권장 범위

첫 스프린트는 아래까지만 잡는 것을 권장한다.

- 로비에 편성 UI 진입 지점 만들기
- 새 전투 씬 빈 골격 만들기
- 에너지 HUD + 유닛 슬롯 HUD 띄우기
- 플레이어당 더미 유닛 3기 편성 데이터 로드
- 코어 1개와 적 압박 더미 1종 배치

## 위험요소

- `Player`와 `Skill` 피처가 현재 구형 규칙을 너무 많이 품고 있어 부분 수정이 누적되면 오히려 느려질 수 있다.
- `JG_GameScene`를 직접 고치기 시작하면 구형/신규 UI와 bootstrap 순서가 충돌할 가능성이 높다.
- 네트워크 동기화 키를 구형 의미로 재사용하면 late-join 복구에서 해석 충돌이 날 수 있다.

이 문서는 그래서 처음부터 **병렬 씬 -> 컷오버 -> 정리**를 기본 전략으로 둔다.
