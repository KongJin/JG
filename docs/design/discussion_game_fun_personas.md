# 재미 vs 리스크 — 4인 전문가 페르소나 토의

워크숍·에이전트 롤플레이용 보조 문서다. **게임 컨셉·수치·MVP 정의의 SSOT는 [game_design.md](../../agent/game_design.md)** 만 따른다. 여기서는 질문·입장 초안·토의 기록만 정리한다.

**근거 요약:** MVP는 2~4인, 약 5분, 웨이브 생존, 프렌들리 파이어, 슬롯 2 + 덱 순환, 웨이브 종료마다 3지선다, 다운 후 구조.

---

## 4인 페르소나 (재미 신호 vs 망할 수 있는 지점)

### A. 코옵·팀 다이내믹스 (Social / Co-op)

- **관점:** 팀이 서로를 **책임지게** 만드는가, **비난**으로 가는가.
- **재미 쪽 논거:** 다운+구조, FF 50% 감쇠, “내가 눕혔는데 내가 살린다” 톤 (`game_design.md` 사망 처리·톤).
- **리스크 논거:** 4인 후반 **투사체 폭증 → 아군 피해가 주연**이 되면 팀이 흩어져 코옵감 붕괴(문서의 프렌들리 파이어 스케일링 주의).
- **토의 질문 예:** “몇 판째부터 ‘아군이 문제’라고 느끼나?” “2인 vs 4인 밸런스를 같은 룰로 가져갈 것인가?” “로비 옵션(난이도 등)이 팀 기대를 어떻게 바꾸나?”

### B. 빌드·런 정체성 (Build / Roguelite)

- **관점:** 한 판 안에서 **빌드가 굵어지는 체감**과 **매판 다른가**.
- **재미 쪽 논거:** 개수/범위/지속 + “새 스킬 vs 강화” 트레이드오프, 웨이브=성장 트리거(경험치 없음).
- **리스크 논거:** **시너지 Phase 2**라 MVP가 **얇은 덱빌딩**에 머물면 반복 플레이 동기가 **메타 성장(상세 미정)** 에만 기대게 됨.
- **토의 질문 예:** “8스킬 풀만으로 4회 선택이 ‘매번 다르다’고 느껴지나?” “메타 없이 10판 버틸 수 있나?” “난이도 프리셋이 빌드 선택의 의미를 바꾸나?”

### C. 전투 UX·온보딩 (Combat feel / Cognitive load + 첫 세션 가독성)

- **관점:** 실시간 카오스에서 **슬롯 2 + 덱 순환**이 읽히는가; **첫 30초**에 왜 아픈지·무엇을 할지 전달되는가.
- **재미 쪽 논거:** 조준 방향 수동 + 타겟 보조만, “가벼운 판단” 목표, 웨이브 중 HUD 단순; 짧은 런·명확한 웨이브 리듬.
- **리스크 논거:** `game_design.md`가 **덱 순환 실시간 UX를 최우선 검증**이라고 명시 — 여기서 실패하면 나머지 기둥이 무너짐. Photon+WebGL에서 설명 실패 시 이탈.
- **토의 질문 예:** “피킹 10초 + 전투 중 순환 읽기”가 동시에 가능한가? 튜토리얼 없이 웨이브1 목표(오폭 학습)가 과하지 않은가? “한 줄 훅을 로비/첫 웨이브에 어떻게 걸 것인가?” 옵션·설정이 늘 때 HUD 부담은?

### D. 밸런스·페이싱·콘텐츠 믹스 (Numbers + 세션·재도전 리듬)

- **관점:** 숫자·웨이브 길이·**적 구성**이 **재미(긴장·웃음)** 를 만들지 **짜증**을 만들지; **재시작·지인 초대 루프**와 맞는가 (`developer_context.md` 배포·검증 루프와 연계).
- **재미 쪽 논거:** Safety를 **세금 옵션 금지**, 공격/범위와 교환 설계 원칙; 빠른 재시작, 세션 길이와 웨이브 리듬.
- **리스크 논거:** FF 감쇠·후반 조정이 미세하면 “실수 vs 운 나쁨” 경계 붕괴; 자동 선택이 **첫 옵션 고정**이라 밸런스 실수 시 **패널티가 구조적으로 고정**됨(UI 리듬). 메타 미정은 장기 리텐션 공백.
- **토의 질문 예:** “첫 선택지 자동 선택이 항상 공정한가?” “웨이브4 이후 감쇠 35~40%는 언제 확정할 것인가?” “적 1종+링 스폰만으로 포지셔닝 재미를 검증할 수 있는가?” “로비 난이도·맵(프리셋)을 어디까지 넣을 것인가?”

## 5인 → 4인 매핑 (참고)

기존 5인 초안을 합치지 않고 **축만 흡수**했다. 유지·리텐션(구 5번)은 없앤 것이 아니라 C·D에 나뉘어 있다.

| 4인 역할 | 흡수하는 기존 5인 |
|----------|-------------------|
| A. 코옵·팀 다이내믹스 | 구 1. 코옵·멀티플레이 |
| B. 빌드·런 정체성 | 구 2. 시스템·로그라이트 |
| C. 전투 UX·온보딩 | 구 3. 전투 UX + 구 5의 첫 30초·가독성·훅 |
| D. 밸런스·페이싱·콘텐츠 믹스 | 구 4. 난이도·밸런스 + 구 5의 세션·재도전·리텐션 공백 |

---

## 토의 운영 루틴 (약 40~45분)

**사전 읽기 (토의 전):** [game_design.md](../../agent/game_design.md)의 MVP·코어 재미, 직전 플레이테스트 메모([playtest_mvp_template.md](playtest_mvp_template.md) 양식 권장).

1. **1라운드 (약 15분):** A → B → C → D 순. 각자 **약 2분** 안에  
   - 가장 강한 **재미 한 줄**  
   - 가장 큰 **리스크 한 줄**  
   - **이번 스프린트에서 시험할 가설 한 줄**
2. **교차 질문 (약 15분):** 미리 정한 **긴장축 2~3개**만 깊게 (위 질문 풀에서 선택).
3. **합의 (약 10분):** `game_design.md`의 **MVP 검증 포인트** 중 **다음 기간에 측정·구현에 태울 1~2개**만 고른다.
4. **기록:** 아래 **시뮬레이션 토론 결과 메모**에 날짜와 함께 추가한다. **바인딩한 수치·MVP 문장 변경은 `game_design.md`에만** 반영한다.

**에이전트 채팅에서:** 역할 A~D 프롬프트를 한 번에 두고 1라운드 형식으로 답하게 한 뒤, 사용자가 긴장축 2~3개를 지정하면 교차 질문 라운드를 시뮬레이션한다. 산출은 **메모 초안**과 **`game_design.md`에 옮길 변경 후보 목록**으로 나누고, SSOT는 사람이 확정한다.

## 샘플 토의 전문 (예시)

아래는 **가상의 한 차례 토의**를 문서에 넣어 둔 것이다. 실제 레포 상태(적 1종·웨이브 테이블이 디자인 5웨이브와 어긋날 수 있음, 덱 순환 UX가 최우선 검증이라는 `game_design.md` 서술)를 전제로 했다. **수치·MVP 문장의 확정은 여전히 `game_design.md`에서만** 한다.

### 전제 (사전 읽기 요약)

- 코어는 혼돈 속 협력, 빌드 성장, 슬롯 2 + 덱 순환, 웨이브마다 3지선다.
- 구현과 디자인 사이에 **적 다양도·웨이브 수** 갭이 있을 수 있다는 점을 D가 의제로 올린다.

---

### 1라운드 — 각자 2분 분량

**A:** 재미 한 줄은 “아군 스킬에도 깎이지만 같이 살린다”는 **책임감 코옵**이에요. 리스크는 후반에 투사체만 보이면 **서로 멀어지는** 거죠. 가설은 “로비에서 난이도만 바꿔도 **기대 밀도**가 달라져서 FF 체감이 먼저 드러난다.”

**B:** 재미는 **웨이브 끝 선택**에서 ‘새 카드 vs 강화’가 매판 갈린다는 점이에요. 리스크는 풀이 얇으면 세 번째 판부터 **같은 느낌**이 난다는 거고요. 가설은 “난이도가 올라가면 **강화 쪽 가치**가 상대적으로 커지는지 본다.”

**C:** 재미는 **조준은 내가, 맞을 적은 시스템이 도와준다**는 분업이에요. 리스크는 피킹 10초와 전투 중 **다음 카드 읽기**가 동시에 걸리면 뇌가 터져요. 가설은 “로비 옵션을 늘리기 전에 **덱/다음 드로우 UI**만으로도 첫 세션 이탈이 줄어드는지 재현 테스트한다.”

**D:** 재미는 **링 스폰 압박**이 단순한데 숫자로 숨이 막히는 순간이에요. 리스크는 적이 한 종류면 `game_design.md`에 있는 **웨이브 2~3 목적**(포지셔닝·역할)을 **데이터만으로는 증명 못 한다**는 거예요. 가설은 “`EnemyData` 하나 더 + 웨이브 엔트리만 바꿔도 **검증 질문이 살아난다**.”

**A:** D 말에 동의하는데, 적만 늘리면 밀도 올라가서 **제가 걱정하는 FF 쪽**이 먼저 터질 수 있어요.

**D:** 그래서 첫 스텝은 **체력·속도만 다른 변종**으로 시작하고, 스폰 수는 난이도 프리셋에서 묶는 게 낫죠.

---

### 교차 질문 — 긴장축 3개

**긴장축 ①:** “적 다양화 vs 덱 UX 검증 순서”

**C → D:** 적 타입 늘리는 건 제가 반대는 아닌데, **HUD에 무엇을 더 넣지 않을지**부터 합의해야 해요. 웨이브 목표 문구라도 늘면 피킹 타이머와 경쟁해요.

**D → C:** 동의. 1차는 **기존 UI 안에서** “느림/튼튼 vs 빠름/약함”만으로 포지셔닝 차이가 나는지 보죠. 별도 튜토리얼 문자열은 다음 스프린트.

**B:** 저는 빌드 입장에선 **새 적 패턴이 생기면** “범위 vs 개수” 선택 의미가 커져서 좋아요. 다만 스킬 풀 고정이면 **난이도만 올리면** 매판 같은 빌드로 수렴할 수 있어요—그건 플레이 테스트로 확인.

**A:** 적이 두 갈래로만 달라도 “누가 빠른 애 잡을지” **말이 나오기 시작**해요. 그게 코옵 신호예요.

**긴장축 ②:** “로비 난이도·맵 프리셋을 언제 넣나”

**A:** 난이도는 **방장만 고르고 전원 동의** UI는 나중에 가도 돼요. 일단 **한 방 안 기대치 통일**이 더 중요해요.

**D:** 데이터는 **난이도 = 스폰 배율 + (선택) 웨이브 길이 배율** 한 줄로 시작하자고요. 맵은 **같은 아레나에 장애물 프리셋** 정도가 현실적이에요.

**C:** 로비에 드롭다운 두 개 생기면 **첫 30초 설명**에 “이 방은 어떤 판인가”가 필요해요. 한 줄이라도 **방 제목이나 배지**로 보여주기로 할까요?

**B:** 찬성. 안 그러면 B가 말한 **빌드 기대**와 방 설정이 안 맞아요.

**긴장축 ③:** “FF 스케일링을 정식 지표에 넣을지”

**A:** 정식 지표로 올리면 좋지만 **측정이 무겁**어요. 질문지 몇 개로 4인 플레이 한 세션씩만 잡자고요.

**D:** 수치 튜닝은 웨이브·적 믹스가 잡힌 뒤가 맞아요. 지금은 **밀도 가설**만 적어 두죠.

**C:** 동의. FF는 **① 덱 UX 통과 후** 스프린트 체크리스트로 넘기자는 예전 메모와 같은 선이에요.

---

### 마무리 발언 (한 줄씩)

**A:** 코옵 검증은 **밀도·역할 대화**가 나오는지 보면 된다.

**B:** 빌드 검증은 **같은 난이도에서도 선택이 갈리는지** 본다.

**C:** 최우선은 **전투 중 덱 읽기 + 첫 웨이브 오폭 학습**이 과하지 않은지.

**D:** 콘텐츠는 **적 변종 1개 + 웨이브 테이블 정렬**을 먼저, 로비 프리셋은 **표시만큼은 단순**하게.

---

### 합의된 결론 (샘플)

다음은 **이번 가상 토의에서 합의로 정리한 결론**이다. 레포에 반영할 때는 **`game_design.md`에 옮길 항목과 구현 백로그를 분리**해서 처리한다.

1. **이번 기간 정식 지표 (2개)**  
   - ① **슬롯 2 + 덱 순환**이 전투 중에 읽히는가(다음 드로우·순환 리듬).  
   - ② **새 스킬 도입 vs 기존 강화**가 매판 트레이드오프로 느껴지는가.

2. **정식 지표 바깥 (병행)**  
   - **프렌들리 파이어·4인 밀도**는 짧은 포커스 인터뷰·체크리스트로 질적 추적. 수치 확정은 적 믹스·웨이브 길이가 한 박자 잡힌 뒤.

3. **구현·검증 순서 (합의)**  
   - 먼저 **①을 플레이테스트로 통과**시키는 데 리소스를 둔다.  
   - 동시에 **적 `EnemyData` 변종 1개와 웨이브 엔트리 구성**으로 `game_design.md`의 웨이브 목적(포지셔닝 압박)에 가까운 **최소 콘텐츠 갭**을 줄인다.  
   - **로비 난이도(및 가능하면 맵/프리셋)** 는 **배율 데이터 + 방에 보이는 한 줄 설명**까지를 1차 범위로 두고, UI는 단순 유지.

4. **`game_design.md`에 옮길 때 적을 문장 예시 (초안일 뿐)**  
   - “MVP 검증 우선순위: 덱 순환 UX → (병행) 적 최소 2 archetype으로 웨이브 목적 검증 → 로비 난이도 프리셋은 표기·배율 단순화.”  
   → **실제 문장·수치는 프로젝트 오너가 SSOT에 반영한다.**

## 시뮬레이션 토론 결과 메모 (기록용)

에이전트 롤플레이 등으로 나온 **가상 합의**다. 실제 측정·수치 변경은 `game_design.md`만 고친다.

### 메모 양식 (복사해 사용)

```text
- 날짜 (YYYY-MM-DD):
- 참가 역할: A / B / C / D (또는 에이전트 롤플레이)
- 이번 기간 정식 지표 (1~2개):
- 교차 질문으로 정한 긴장축 (2~3개):
- 가설·다음 액션 (구현·플레이테스트):
- game_design.md에 옮길 바인딩 변경 후보 (있으면):
```

### 과거 메모

- **(초기 기록)** **이번 분기 정식 지표 2개:** ① 슬롯 2 덱 순환 직관, ② 새 스킬 vs 강화 고민.  
- **⑤ FF:** 정식 지표 밖. 4인 내 플레이·짧은 포커스 인터뷰로 질적 병행.  
- **③·④:** ① 소규모 통과 후 스프린트 체크리스트.  
- **(샘플 토의)** 전문 대화·합의는 위 **`## 샘플 토의 전문 (예시)`** 를 본다. 별도 날짜 메모를 남길 때는 그 절의 **합의된 결론**을 요약해 여기에 한 블록 추가하면 된다.

## 부록: 기존 5인 상세 (질문 풀·역사)

아래는 4인으로 재구성하기 전의 **5인** 상세다. 토의에서 더 세분된 질문이 필요할 때 참고한다.

### 1. 코옵·멀티플레이 디자이너 (Social / Co-op)

- **관점:** 팀이 서로를 **책임지게** 만드는가, **비난**으로 가는가.
- **재미 쪽 논거:** 다운+구조, FF 50% 감쇠, “내가 눕혔는데 내가 살린다” 톤 (`game_design.md` 사망 처리·톤).
- **리스크 논거:** 4인 후반 **투사체 폭증 → 아군 피해가 주연**이 되면 팀이 흩어져 코옵감 붕괴(문서의 프렌들리 파이어 스케일링 주의).
- **토론 질문:** “몇 판째부터 ‘아군이 문제’라고 느끼나?” “2인 vs 4인 밸런스를 같은 룰로 가져갈 것인가?”

### 2. 시스템·로그라이트 디자이너 (Build / Run identity)

- **관점:** 한 판 안에서 **빌드가 굵어지는 체감**과 **매판 다른가**.
- **재미 쪽 논거:** 개수/범위/지속 + “새 스킬 vs 강화” 트레이드오프, 웨이브=성장 트리거(경험치 없음).
- **리스크 논거:** **시너지 Phase 2**라 MVP가 **얇은 덱빌딩**에 머물면 반복 플레이 동기가 **메타 성장(상세 미정)** 에만 기대게 됨.
- **토론 질문:** “8스킬 풀만으로 4회 선택이 ‘매번 다르다’고 느껴지나?” “메타 없이 10판 버틸 수 있나?”

### 3. 전투 UX·입력 디자이너 (Combat feel / Cognitive load)

- **관점:** 실시간 카오스에서 **슬롯 2 + 덱 순환**이 읽히는가.
- **재미 쪽 논거:** 조준 방향 수동 + 타겟 보조만, “가벼운 판단” 목표, 웨이브 중 HUD 단순.
- **리스크 논거:** `game_design.md`가 **덱 순환 실시간 UX를 최우선 검증**이라고 명시 — 여기서 실패하면 나머지 기둥이 무너짐.
- **토론 질문:** “피킹 10초 + 전투 중 순환 읽기”가 동시에 가능한가? 튜토리얼 없이 웨이브1 목표(오폭 학습)가 과하지 않은가?

### 4. 난이도·경제 밸런스 (FF / Safety / Numbers)

- **관점:** 숫자가 **재미(긴장·웃음)** 를 만들지 **짜증**을 만들지.
- **재미 쪽 논거:** Safety를 **세금 옵션 금지**, 공격/범위와 교환 설계 원칙.
- **리스크 논거:** FF 감쇠·후반 조정이 미세하면 “실수 vs 운 나쁨” 경계 붕괴; 자동 선택이 **첫 옵션 고정**이라 밸런스 실수 시 **패널티가 구조적으로 고정**됨(UI 리듬).
- **토론 질문:** “첫 선택지 자동 선택이 항상 공정한가?” “웨이브4 이후 감쇠 35~40%는 언제 확정할 것인가?”

### 5. 유지·리텐션·라이브 운영 (Session / Onboarding / WebGL)

- **관점:** 5분 세션과 **지인 초대·재도전 루프**가 맞는가 (`developer_context.md`의 배포·검증 루프와 연계 가능).
- **재미 쪽 논거:** 짧은 런, 빠른 재시작 흐름, 명확한 웨이브 리듬.
- **리스크 논거:** Photon+WebGL에서 **첫 30초**에 “왜 아픈지” 설명 실패 시 이탈; 메타 상세 미정은 **장기 리텐션 공백**.
- **토론 질문:** “첫 판 이탈 지점은 어디로 잡나?” “한 줄 훅을 로비/첫 웨이브에 어떻게 걸 것인가?”
