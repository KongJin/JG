# /agent/codex_patching.md

> 마지막 검토: 2026-04-11

이 문서는 **Codex 전용** 패치 운영 규칙의 SSOT다.
사람 개발자의 일반 Git workflow 전체를 정의하지 않으며, **Codex가 repo-tracked 파일을 수정할 때** `apply_patch` 운영 방식만 다룬다.

전역 구조 규칙은 `architecture.md`, 문서 소유권과 응집도 원칙은 `work_principles.md`, Unity 고유 직렬화 규칙은 `unity.md`가 소유한다.
이 문서는 **패치 분할, 적용 순서, 패치 직후 검증 루프**만 정의한다.

---

## 목적

- 큰 패치를 한 번에 넣다가 도구 입력 길이/적용 실패로 무너지는 일을 줄인다.
- 한 번 실패했을 때 어디까지 적용됐는지 애매해지는 상황을 줄인다.
- 변경을 작은 책임 단위로 나눠, 중간 검증과 diff 읽기를 쉽게 만든다.

---

## 기본 원칙

## 1. 한 패치 = 한 책임

한 번의 `apply_patch`는 하나의 논리 변경만 담는다.

- 새 타입 추가
- 기존 참조 갱신
- 문서 동기화
- 씬 생성기/에디터 도구 수정

위 항목을 한 패치에 많이 섞지 않는다.

## 2. 패치가 커지면 먼저 나눈다

아래 조건 중 하나라도 걸리면 기본값은 **패치 분할**이다.

- 바뀌는 파일이 `5개 이상`
- 새 파일이 많다
- 큰 `Add File`과 큰 `Update File`이 함께 있다
- 패치 본문이 대략 `200줄`을 넘는다
- `Delete File` 후 큰 재생성을 같이 넣고 있다

예외는 허용하지만, 예외를 기본값처럼 쓰지 않는다.

---

## 파일 종류별 기본값

## 새 파일 추가

- 한 패치에 `2~4개`까지만 추가한다.
- 새 파일 총합이 크면 `파일당 1패치` 또는 `2파일 1패치`로 더 잘게 자른다.
- 새 타입 여러 개가 같은 변경이라도, 참조 대상 파일 수정은 다음 패치로 미룬다.

## 기존 파일 수정

- 한 패치에 `1~3개`까지만 수정한다.
- 같은 타입명을 여러 파일에서 따라 바꾸는 경우도 먼저 핵심 파일부터 나누어 넣는다.

## 삭제 후 재생성

- 큰 클래스 교체는 `Delete File`과 `Add File`을 별도 패치로 나눈다.
- 삭제 직후 새 파일을 다시 넣더라도, 한 패치에 모두 욱여넣지 않는다.

## 코드 vs 문서

- 코드 수정과 문서 수정은 같은 작업이라도 기본적으로 별도 패치로 나눈다.
- 문서만 바꾸는 패치는 마지막 정리 단계에 둔다.

---

## 권장 적용 순서

Codex는 기본적으로 아래 순서를 따른다.

1. 새 타입/새 파일 추가
2. 기존 참조 파일 수정
3. 씬 생성기, 에디터 도구, wiring 코드 수정
4. 문서 동기화

이 순서를 지키면, 중간 상태에서 실패하더라도 무엇이 아직 안 붙었는지 찾기 쉽다.

---

## 패치 직후 검증 루프

각 패치 직후 바로 가벼운 검증을 한다.

- `rg`로 새 심볼 참조 상태 확인
- 필요한 파일만 `Get-Content`로 재확인
- 가능하면 작은 단위 빌드 또는 정적 점검 실행

검증 없이 다음 큰 패치로 넘어가지 않는다.

특히 아래 상황에서는 **패치마다 바로 확인**한다.

- rename 또는 symbol move 직후
- Setup/Bootstrap/scene wiring 수정 직후
- 새 문서를 SSOT로 연결한 직후
- 긴 패치를 둘 이상 연속으로 넣을 때

---

## 예외 규칙

기계적 치환처럼 원자성이 더 중요한 경우는 예외를 허용한다.
단, 예외를 쓸 때는 먼저 아래를 정한다.

- 대상 패턴
- 제외 대상
- 검증 방법

이 사전 정리 없이 큰 패치를 한 번에 넣지 않는다.

---

## 하지 말아야 할 것

- 큰 `Add File` 여러 개와 큰 `Update File` 여러 개를 한 패치에 동시에 넣기
- 실패한 대형 패치를 거의 그대로 다시 재시도하기
- 패치 직후 검증 없이 다음 패치로 넘어가기
- 문서와 코드를 한 패치에 길게 섞어 넣기
- “이번엔 되겠지” 식으로 입력 길이를 밀어붙이기

---

## 빠른 체크리스트

패치 전에 아래를 확인한다.

1. 이번 패치는 한 책임만 담고 있는가?
2. 파일 수가 많으면 더 나눌 수 있는가?
3. 새 파일과 참조 수정이 섞여 있으면 분리했는가?
4. 삭제와 재생성을 따로 넣었는가?
5. 패치 직후 어떤 명령으로 검증할지 정했는가?

확실하지 않을 때는 **더 작은 패치**를 선택한다.
