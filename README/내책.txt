1. 제목

많은 설계 원칙이 가리키고 있던 방향

부제
코드 변경이 어려워지는 이유

2. 목차

코드 변경이 어려워지는 이유

책임과 SRP

응집도와 캡슐화

결합도와 의존 방향

정리

3. 문제 제기

코드를 수정하려고 하면
생각보다 일이 커지는 경험이 있다

• 수정하려고 파일을 계속 찾아다닌다
• 하나 고쳤는데 다른 곳이 깨진다
• 수정이 제대로 되었는지 확신하기 어렵다

4. 개발자 밈

why is this code everywhere

fix one bug break three things

이 밈은 단순한 농담이 아니라
코드 구조에서 실제로 발생하는 문제를 보여준다

5. 우리는 많은 것을 배운다

코드를 관리하기 위해 많은 개념을 배우게 된다

• 안티패턴
• 설계 원칙
• 리팩토링 가이드
• 아키텍처 / 디자인 패턴

하지만 이런 질문이 생긴다

언제 어떤 원칙을 적용해야 할까?

6. 공통 질문

많은 설계 문제는
결국 같은 질문으로 이어진다

왜 관련된 코드가 한곳에 모여 있지 않을까

왜 변경이 여러 곳으로 퍼질까

이 질문을 설명하는 두 가지 개념

응집도
결합도

7. 책임을 판단하는 방법

우리는 보통
주제나 기능으로 책임을 판단한다

SkillManager

데미지 계산
이펙트 출력

둘 다 스킬 기능이기 때문에
응집도가 높은 설계처럼 보인다

8. SRP

A class should have only one reason to change

책임은 기능이 아니라
바뀌는 이유(reason to change) 로 판단한다

예

데미지 계산 → 게임 밸런스 변경

이펙트 출력 → 연출 변경

9. 응집도

응집도는

같은 이유로 바뀌는 코드를 한곳에 모으고
다른 이유로 바뀌는 코드는 분리하는 것

응집도가 낮을 때 나타나는 징후

• God Class
• Shotgun Surgery
• Anemic Domain Model

10. 캡슐화

캡슐화는

변경을 내부에 가두는 기술이다

캡슐화를 통해

• 관련된 변화는 내부에 모이고
• 외부로는 변경 영향이 퍼지지 않는다

즉

캡슐화는
응집도를 실제 구조로 만드는 방법이다

11. 결합도

결합도는

한 모듈의 변경이
다른 모듈의 변경을 요구하지 않게 하는 것

결합도가 높을 때 나타나는 징후

• 하나 수정하면 여러 곳에서 오류 발생
• 작은 수정이 예상치 못한 부작용 발생
• 코드를 떼었다 붙이기 어렵다

12. 의존 방향

결합도를 낮추는 핵심은
의존 방향을 제어하는 것

DIP

구현이 아니라
추상화에 의존한다

ADP

의존 그래프에 순환이 없어야 한다

13. 실무 규칙

의존성을 명확하게 드러내라

• 의존성은 생성자나 파라미터로 전달
• DI 시점에 유효성 검사 (Fail Fast)

숨겨진 의존이 줄어들수록
결합도는 낮아진다

14. 정리

많은 설계 원칙을 살펴보면
결국 같은 방향을 가리키고 있다

응집도

변경 이유를 한곳에 모은다

결합도

변경 영향이 다른 곳으로 퍼지지 않게 한다

캡슐화

변경을 내부에 가두는 기술이다

마지막 슬라이드 (강하게 끝내기)

좋은 설계는

변경을 예측하는 것이 아니라

변경을 견디는 구조를 만드는 것이다
