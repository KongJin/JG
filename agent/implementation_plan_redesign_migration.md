# /agent/implementation_plan_redesign_migration.md

## 목적

이 문서는 현재 구현을 새 게임 방향에 맞춰 옮기기 위한 migration 계획서다.
제품 SSOT는 `agent/game_design.md`이며, 이 문서는 설계를 다시 정의하지 않고 구현 전환 순서와 갭만 정리한다.
실제 작업 순서, 씬 교체 전략, 컷오버 체크포인트는 `agent/implementation_plan_redesign_execution.md`를 따른다.

---

## 확정된 개념 전환

| 구형 개념 | 신규 개념 |
|---|---|
| 직접 조작 Player | 차고 소유자 + 전투 중 소환자 |
| Mana 기반 스킬 사용 | Energy 기반 유닛 소환 |
| Skill deck rotation | Garage unit roster + battle slot |
| 웨이브 사이 보상 선택 | 매치 전 편성 중심 |
| 프렌들리 파이어/다운 구조 | 공용 코어 방어 + 연속 압박 대응 |

## 현재 구현과의 대표 갭

현재 코드베이스의 피처 이름과 구조는 여전히 구형 액션 루프를 반영한다.
당분간 폴더 이름이 남아 있더라도, 신규 작업 판단은 기능 의미를 새 방향 기준으로 해석한다.

### 주요 영향 범위

- `Assets/Scripts/Features/Player/`
  - 직접 조작 아바타 중심 책임이 남아 있을 가능성이 높다.
- `Assets/Scripts/Features/Skill/`
  - 직접 시전 스킬, 스킬 슬롯, 스킬 표현 자산 중심 구조를 유닛/모듈 방향으로 재정의해야 한다.
- `Assets/Scripts/Features/Combat/`
  - 플레이어 액션 판정 중심 전투를 소환/배치/앵커 반경 교전 중심으로 바꿔야 한다.
- `Assets/Scripts/Features/Wave/`
  - 웨이브 보상 선택 중심 흐름이 남아 있다면 연속 압박 기반 구조로 옮겨야 한다.
- `Assets/Scripts/Features/Enemy/`
  - 적 개체 자체보다 압박 조합과 역할 분류를 먼저 맞춰야 한다.
- `Assets/Scripts/Features/Lobby/`
  - 로비 출발점은 재사용 가능하지만, 시작 전 편성 진입 흐름이 추가되어야 한다.

## 전환 원칙

1. 제품 개념은 먼저 바꾸고, 구현은 단계적으로 따라온다.
2. 구형 시스템을 신규 시스템 위에 임시로 덧씌우지 않는다.
3. 같은 의미를 구형/신규 용어로 이중 정의하지 않는다.
4. 씬 계약과 feature README는 실제 wiring이 바뀌는 시점에 함께 갱신한다.

## 권장 작업 순서

### 1. 용어와 루프 고정

- `agent/game_design.md`를 기준으로 신규 용어를 고정한다.
- 신규 문서와 신규 코드에서는 `mana`, `skill deck`, `friendly fire`, `down/rescue`를 제품 핵심으로 다시 도입하지 않는다.
- 팀 공용 용어는 `garage`, `unit`, `module`, `energy`, `summon`, `core defense`, `anchor radius`로 통일한다.

### 2. 게임 진입 플로우 전환

- 로비 다음 단계에 `차고/편성` 단계를 넣는다.
- 전투 진입 조건을 "시작 스킬 선택 완료"가 아니라 "편성 완료"로 바꾼다.
- 결과 화면 이후 재도전 시 편성 재검토가 자연스럽게 이어지도록 흐름을 잡는다.

### 3. 전투 입력 전환

- 직접 조작 이동/공격 입력보다 유닛 슬롯 선택과 배치 입력을 우선 구현한다.
- 개인 자원은 마나 대신 에너지로 통일한다.
- 전투 중 판단은 "무엇을, 언제, 어디에" 소환하는가로 수렴시킨다.

### 4. 전투 오브젝트 책임 재정의

- Player는 월드 아바타보다 플레이어 소유 상태와 소환 권한을 관리하는 방향으로 줄인다.
- Skill은 직접 시전 기술보다 유닛 모듈, 유닛 능력, 유닛 프리젠테이션 자산으로 재편한다.
- Combat은 유닛 배치, 앵커 반경, 자동 교전 규칙을 중심으로 재구성한다.

### 5. 전투 구조 전환

- 웨이브 보상 선택 UI와 로직은 우선순위를 내린다.
- 공용 코어 방어와 연속 압박 대응을 전투 기본 구조로 옮긴다.
- 첫 버전에서는 재배치, 웨이포인트, 고급 AI를 넣지 않는다.

### 6. 콘텐츠 최소 세트 구성

- 반개방 맵 1종
- 공용 코어 1종
- 적 역할 3~4종
- 플레이어당 유닛 편성 3~5기
- 유닛 역할 차이를 만드는 최소 모듈 조합

### 7. 문서/피처 README 동기화

- 각 feature가 새 책임으로 실제 전환되면 해당 README를 같은 패스에서 갱신한다.
- 초기화 순서가 바뀌면 `agent/initialization_order.md`를 갱신한다.
- 피처 경계가 재편되면 `agent/architecture.md`를 갱신한다.

## 임시 허용 상태

- 기존 feature 폴더명 `Player`, `Skill`, `Wave`가 당장 남아 있는 것
- 일부 구형 프리젠테이션 자산을 신규 유닛 표현에 재활용하는 것
- 로비/룸 시작 구조를 유지한 채 편성 단계를 사이에 삽입하는 것

허용하지 않는 것은 구형 게임 재미를 SSOT처럼 되살리는 임시 타협이다.

---

## 완료 기준

- 문서상 주 게임 플로우가 로비 -> 편성 -> 전투 -> 결과로 통일되어 있을 것
- 신규 구현에서 플레이어 기본 입력이 유닛 소환 중심일 것
- 신규 자원명이 에너지로 통일되어 있을 것
- MVP 범위가 공용 코어 방어 + 편성 기반 코옵 PvE로 읽힐 것
- 구형 액션 코옵 규칙이 SSOT 문서에서 제거되어 있을 것
